| 항목 | Phase 1 포함? | 사유 |
|---|---|---|
| PG 인덱스 7개 추가 | ✅ 포함 | 무위험, 즉시 효과(월말 집계 10배 가속) |

## 🚦 P0 — PG 인덱스 추가 (핵심, 약 5분)

### P0-1. 서버 접속 & 자료 폴더 최신화
**무엇**: 운영서버 SSH 접속 후 최신 자료 파일 pull
**어디서**: 로컬 PC → ssh → `root@223.130.131.188`
**어떻게**:
```bash
ssh root@223.130.131.188
cd /opt/zeroda-platform
git pull origin main
ls zeroda_platform/인프라증설_Phase1_자료/
```
**왜**: 자료 4개 파일이 서버에 실제로 존재해야 다음 단계 실행 가능
**검증**: `ls` 결과에 `01_pg_index_add.sql` 포함 4개 파일이 보여야 함
**실패 시**:
- `git pull` 에러 → `git fetch && git reset --hard origin/main`
- 파일 없음 → 사장님께 보고 (기획 세션에서 git push 누락 가능성)

### P0-2. 인덱스 추가 실행
**무엇**: `01_pg_index_add.sql` 을 psql로 실행
**어디서**: 서버 셸
**어떻게**:
```bash
set -a; source /opt/zeroda-platform/.env; set +a
export PGPASSWORD="$ZERODA_PG_PASSWORD"

psql -h "$ZERODA_PG_HOST" \
     -U "$ZERODA_PG_USER" \
     -d "$ZERODA_PG_DB" \
     -p "${ZERODA_PG_PORT:-5432}" \
     -f /opt/zeroda-platform/zeroda_platform/인프라증설_Phase1_자료/01_pg_index_add.sql
```
**왜**: 기사모드 조회·월말 정산 10배 가속. 이게 Phase 1의 핵심 효과
**검증**: 실행 로그에 `CREATE INDEX` 메시지 7번, `VACUUM` 3번 나와야 함. 에러 메시지 없어야 함.
확인 쿼리:
```bash
psql -h "$ZERODA_PG_HOST" -U "$ZERODA_PG_USER" -d "$ZERODA_PG_DB" \
     -p "${ZERODA_PG_PORT:-5432}" -c "
SELECT tablename, indexname FROM pg_indexes
 WHERE tablename IN ('real_collection','schedules','processing_confirm')
 ORDER BY tablename, indexname;"
```
기대 출력: 최소 7개 `idx_*` 인덱스 + 기존 PK 인덱스

**실패 시**:
| 에러 메시지 | 처치 |
|---|---|
| `relation "real_collection" does not exist` | PG가 아닌 SQLite 모드일 가능성 → `.env` 재확인, 사장님 보고 |
| `permission denied` | zeroda 사용자 권한 부족 → NCloud 콘솔에서 권한 확인 |
| `could not extend file` (디스크 부족) | `/data` 디스크 여유 공간 확인 (`df -h`) |
| `relation "idx_xxx" already exists` | 정상 (`IF NOT EXISTS`로 스킵됨), 무시 가능 |

### P0-3. Reflex 서비스 상태 확인 (무재시작 확인)
**무엇**: 인덱스 추가는 서비스 재시작 불필요. 현재 세션 영향 없는지 확인
**어디서**: 서버 셸
**어떻게**:
```bash
systemctl status zeroda-reflex | head -15
journalctl -u zeroda-reflex -n 20 --no-pager
```
**왜**: 인덱스 추가 중 앱에서 타임아웃이나 락 대기가 없었는지 확인
**검증**: `Active: active (running)` 상태, 최근 로그에 ERROR 없음
**실패 시**: ERROR가 있으면 `04_rollback.sql` 즉시 실행 → 사장님 보고

## 🚦 P1 — pg_stat_statements 활성화 (선택, 약 3분)

### P1-1. 확장 활성화 시도
**무엇**: `02_pg_slow_query_enable.sql` 실행
**어디서**: 서버 셸
**어떻게**:
```bash
psql -h "$ZERODA_PG_HOST" -U "$ZERODA_PG_USER" -d "$ZERODA_PG_DB" \
     -p "${ZERODA_PG_PORT:-5432}" \
     -f /opt/zeroda-platform/zeroda_platform/인프라증설_Phase1_자료/02_pg_slow_query_enable.sql
```
**왜**: Phase 2(Redis) 전후 성능 비교를 위한 베이스라인 데이터 수집
**검증**: `pg_extension` 조회 결과에 `pg_stat_statements` 1행 존재

**실패 시**:
| 에러 | 처치 |
|---|---|
| `shared_preload_libraries must be set` | NCloud 콘솔 → DB파라미터 그룹 → `shared_preload_libraries` 에 `pg_stat_statements` 추가 → DB 재시작 → 재실행. **재시작은 사장님 승인 후에만** |
| `permission denied to create extension` | 관리자 권한 필요 → 스킵 가능 (P1은 선택사항). 사장님께 보고 후 Phase 2 때 함께 처리 |

> ⚠ P1이 실패해도 P0이 성공했으면 Phase 1 목표는 달성된 것임. P1은 **Nice-to-have**.
